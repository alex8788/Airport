# 機場報到與登機系統 — 專案分析報告

本專案模擬真實機場的離境通關流程，採用高度模組化的物件導向設計。以下為系統架構、背景設定、通關流程與異常攔截機制的完整分析。

---

## 一、 架構設計（Architecture Design）

本專案將機場系統拆分為四大模組，每個類別（Class）都有明確的職責。

### 1. 主控系統與介面（Main System）

#### **系統指揮中心（`AirportSystem`）**
* **說明**：這是整個機場系統的進入點，負責掌控流程順序、建立航班並與使用者互動。
* **當前航班清單（`flights`）**：一個陣列空間，用來儲存當前機場看板上的 3 個航班資訊，確保旅客有實際的飛機可以搭乘。
* **初始化看板（`initializeFlights`）**：系統啟動時會隨機挑選 3 條航線並顯示在螢幕上，營造真實機場氛圍。
* **旅客建檔（`setupPassenger`）**：透過一連串的提問，收集使用者的輸入資料，藉此在系統內打造出一個完整的旅客數位身分。
* **輸入防呆工具（`readNonNegDouble` / `readString`）**：確保使用者不小心輸入錯誤格式（例如重量輸入了文字）時，系統不會當機，而是溫柔地提示並要求重新輸入。

### 2. 實體資料模型（Data Models）

#### **旅客（`Passenger`）**
* **說明**：代表真實世界中的旅客，負責保管該旅客所有的隨身物品。
* **隨身物品（`passport` / `baggage` / `ticket`）**：旅客身上的護照、行李與機票物件，通關時海關與地勤人員會直接檢查這些物品的狀態。

#### **機票（`Ticket`）**
* **說明**：紀錄旅客的搭乘權限與通關進度。
* **座位號碼（`seatId`）**：預設為「未分配」（Unassigned），直到報到櫃檯劃位後，才會記錄旅客的專屬座位。
* **通關狀態（`state`）**：紀錄這張機票目前闖到哪一關，防止旅客跳過安檢直接前往登機門。

#### **航班（`Flight`）**
* **說明**：代表一架準備起飛的客機，負責管理飛機上的座位與乘客名單。
* **已佔用座位（`occupiedSeats`）**：紀錄已經有人坐的座位清單，地勤劃位時必須避開這些號碼。
* **預先佔位（`preOccupySeats`）**：為了讓系統更真實，開放報到前會先隨機塞滿一定數量的座位，模擬其他旅客已經提早報到的情況。
* **列印座位表（`printSeatMap`）**：將飛機目前的空位與佔用狀況，繪製成直觀的「雙走道圖表」顯示在終端機螢幕上。

#### **系統列舉設定（Enums）**
* **機票生命週期（`TicketState`）**：嚴格定義了機票只能處於已預訂（`BOOKED`）、已報到（`CHECKED_IN`）、安檢通過（`SECURITY_CLEARED`）、已登機（`BOARDED`）四種狀態。
* **艙等限制（`CabinClass`）**：定義了經濟艙（`ECONOMY`）、商務艙（`BUSINESS`）、頭等艙（`FIRST`）各自專屬的行李載重上限。

### 3. 機場關卡服務（Service Checkpoints）

#### **機場通關介面（`Processable`）**
* **說明**：這是一個介面（規則約定），規定機場裡所有的「關卡」都必須具備處理旅客（`process`）的能力，讓系統能標準化作業。

#### **三大實體關卡（`CheckInCounter`, `SecurityCheck`, `BoardingGate`）**
* **說明**：分別代表報到櫃檯、安檢站與登機門。每個關卡都遵守了 `Processable` 規則，內部寫有各自獨立的檢查邏輯與驗證機制。

## 二、 背景設計（Context & Assumptions）

為了讓程式跑起來具有真實感，本專案在設計時預設了以下情境：

* **定期航班制**：機場設有固定的航線庫（`Route` 列舉）。系統每次啟動都會從以下 6 條航線中，隨機選出 3 條作為當日航班顯示在看板上：

| 航線編號（`Route`） | 目的地（`destination`） | 飛行時間（分鐘） |
| :--- | :--- | :--- |
| **BR198** | 東京（NRT） | 200 |
| **BR159** | 首爾（ICN） | 150 |
| **CI909** | 香港（HKG） | 100 |
| **CI751** | 新加坡（SIN） | 280 |
| **JX800** | 曼谷（BKK） | 220 |
| **CX421** | 吉隆坡（KUL） | 270 |

* **自動化時間推算**：系統會隨機產生一個合理的登機時間，隨後自動加上 30 分鐘作為起飛時間，並依照上方表格的「飛行時間」自動推算出抵達時間，免去人工輸入的繁瑣。
* **單一旅客視角**：為求系統展示順暢，旅客搭乘的班機強制綁定為電子看板上的「第一班飛機（`flights[0]`）」，以此來推進並展示單一旅客的完整過關體驗。

## 三、 流程演示（Flow Demonstration）

旅客從抵達機場到順利登機，在程式碼中會經歷以下完整的互動流程：

**階段一：系統初始化（Initialization）**
* 程式從主控台 `main()` 啟動，立刻呼叫看板建立方法 `initializeFlights()`。
* 終端機畫面上會印出 3 個當前航班的電子看板，讓旅客知道今天的班機資訊。

**階段二：旅客建檔（Passenger Setup）**
* 系統呼叫設定旅客的方法 `setupPassenger()`，引導使用者輸入姓名、行李重量等個人資訊。
* 在背景中，系統會呼叫航班的 `flight.preOccupySeats(30)`，悄悄為這班飛機隨機劃掉 30 個位子，模擬現場的其他旅客。
* 接著為旅客隨機分配一種艙等 `CabinClass`，並將所有資料打包成一個完整的旅客物件 `Passenger`。

**階段三：報到櫃檯（Check-in Counter）**
* 旅客物件進入地勤報到系統 `CheckInCounter`。
* 系統檢查機票購買者 `ticket.getOwner()` 與本名 `passenger.getName()` 是否一致，然後秤重確認行李沒有超出艙等上限，同時檢查是否攜帶違禁品。
* 檢查通過後，呼叫航班的隨機劃位功能 `flight.assignRandomSeat()`，將機票狀態更新為「已報到」（`CHECKED_IN`），並在畫面上印出專屬的雙走道座位表，用 `[P]` 標示出旅客的位置。

**階段四：安檢管制區（Security Check）**
* 旅客走向安檢站 `SecurityCheck`。
* 系統先檢查機票狀態，確認他已經報到過。接著呼叫 `passenger.hasPassport()` 確認旅客有帶護照，且護照上的名字與本人相符。
* 檢查通過，機票狀態更新為「安檢通過」（`SECURITY_CLEARED`）。

**階段五：登機門（Boarding Gate）**
* 旅客來到最後一關登機門掃描器 `BoardingGate`。
* 系統最後一次驗證機票，確認狀態為 `SECURITY_CLEARED`，代表旅客沒有逃避安檢。
* 驗證成功，狀態變更為「已登機」（`BOARDED`），並呼叫航班的加入名單方法 `flight.addPassenger()`，系統印出最終的航站廣播，通關圓滿結束。

## 四、 異常處理（Exception Handling）

機場的安全不容妥協。只要任何一個關卡發現旅客違規，系統就會直接拋出對應的異常警報（自訂例外處理類別 `AirportException` 家族），強制中斷該名旅客的通關流程，並在畫面上顯示 `[系統攔截]`：

* **身分核對異常（`IdentityException`）**
  * **觸發條件**：若旅客姓名與機票購買人不同（在報到櫃檯觸發），或旅客姓名與護照登記名不同（在安檢站觸發）。
  * **系統行為**：報錯並印出「預期的姓名」與「實際查到的姓名」供地勤人員參考。

* **機票狀態異常（`TicketStateException`）**
  * **觸發條件**：旅客試圖「越級通關」時。例如還沒報到就跑去安檢，或是沒過安檢就想登機。
  * **系統行為**：報錯並精確提示「機票預期應處於什麼狀態」以及「機票目前的違規狀態」。

* **行李違規異常（`BaggageException`）**
  * **觸發條件**：於報到櫃檯時，發現行李含有違禁品，或是重量超出該旅客專屬艙等的負重上限。
  * **系統行為**：若是超重，系統會聰明地印出「當前實際重量」、「該旅客艙等名稱」以及「該艙等的載重上限數字」來解釋 拒絕託運。

* **證件缺失異常（`DocumentException`）**
  * **觸發條件**：旅客走到安檢站，系統卻發現他根本沒有攜帶護照。
  * **系統行為**：報錯並明示旅客未攜帶護照這項重要證件。
